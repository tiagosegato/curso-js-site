Requisitos do Projeto

Especificação detalhada do sistema de gerenciamento para estúdio de pilates

Visão Geral dos Requisitos

O sistema de gerenciamento para estúdio de pilates tem como objetivo principal facilitar o cadastro e gerenciamento de alunos, permitindo um controle eficiente das informações. Abaixo estão detalhados os requisitos funcionais e não funcionais do sistema.

Requisitos Funcionais

Os requisitos funcionais descrevem as funcionalidades que o sistema deve oferecer aos usuários.

RF01 - Gerenciamento de Alunos

  • RF01.1: O sistema deve permitir o cadastro de novos alunos com informações como nome, email, telefone e data de nascimento.
  • RF01.2: O sistema deve permitir a visualização de todos os alunos cadastrados.
  • RF01.3: O sistema deve permitir a visualização detalhada de um aluno específico.
  • RF01.4: O sistema deve permitir a atualização dos dados de um aluno.
  • RF01.5: O sistema deve permitir a exclusão de um aluno.
  • RF01.6: O sistema deve permitir a marcação de um aluno como ativo ou inativo.

RF02 - Validação de Dados

  • RF02.1: O sistema deve validar o formato do email do aluno.
  • RF02.2: O sistema deve validar se o nome do aluno foi preenchido.
  • RF02.3: O sistema deve validar se o email do aluno é único no sistema.

RF03 - API RESTful

  • RF03.1: O sistema deve fornecer endpoints para todas as operações CRUD de alunos.
  • RF03.2: O sistema deve retornar respostas em formato JSON.
  • RF03.3: O sistema deve utilizar os métodos HTTP apropriados para cada operação (GET, POST, PUT, DELETE).
  • RF03.4: O sistema deve retornar códigos de status HTTP apropriados para cada resposta.

Requisitos Não Funcionais

Os requisitos não funcionais descrevem características de qualidade que o sistema deve possuir.

RNF01 - Desempenho

  • RNF01.1: O sistema deve responder a requisições em menos de 1 segundo em condições normais.
  • RNF01.2: O sistema deve suportar pelo menos 100 requisições simultâneas.

RNF02 - Segurança

  • RNF02.1: O sistema deve armazenar dados sensíveis de forma segura.
  • RNF02.2: O sistema deve implementar validação de entrada para prevenir ataques como SQL Injection.

RNF03 - Usabilidade

  • RNF03.1: A API deve ser intuitiva e seguir padrões RESTful.
  • RNF03.2: A API deve fornecer mensagens de erro claras e informativas.

RNF04 - Manutenibilidade

  • RNF04.1: O código deve seguir boas práticas de desenvolvimento e padrões de projeto.
  • RNF04.2: O sistema deve ser modular e facilmente extensível.
  • RNF04.3: O código deve ser bem documentado.

RNF05 - Compatibilidade

  • RNF05.1: A API deve ser compatível com os principais navegadores e clientes HTTP.
  • RNF05.2: O sistema deve funcionar em ambientes Linux e Windows.

Casos de Uso

Os casos de uso descrevem as interações entre os usuários e o sistema.

UC01 - Cadastrar Aluno

Ator Principal: Administrador

Descrição: Este caso de uso permite ao administrador cadastrar um novo aluno no sistema.

Fluxo Principal:

  1. O administrador envia uma requisição POST para o endpoint /api/alunos com os dados do aluno.
  2. O sistema valida os dados recebidos.
  3. O sistema cadastra o aluno no banco de dados.
  4. O sistema retorna os dados do aluno cadastrado com status 201 (Created).

Fluxo Alternativo:

  1. Se os dados forem inválidos, o sistema retorna uma mensagem de erro com status 400 (Bad Request).
  2. Se o email já estiver cadastrado, o sistema retorna uma mensagem de erro com status 409 (Conflict).
UC02 - Listar Alunos

Ator Principal: Administrador

Descrição: Este caso de uso permite ao administrador visualizar todos os alunos cadastrados no sistema.

Fluxo Principal:

  1. O administrador envia uma requisição GET para o endpoint /api/alunos.
  2. O sistema recupera todos os alunos do banco de dados.
  3. O sistema retorna a lista de alunos com status 200 (OK).
UC03 - Buscar Aluno

Ator Principal: Administrador

Descrição: Este caso de uso permite ao administrador visualizar os detalhes de um aluno específico.

Fluxo Principal:

  1. O administrador envia uma requisição GET para o endpoint /api/alunos/{id}.
  2. O sistema busca o aluno com o ID especificado no banco de dados.
  3. O sistema retorna os dados do aluno com status 200 (OK).

Fluxo Alternativo:

  1. Se o aluno não for encontrado, o sistema retorna uma mensagem de erro com status 404 (Not Found).
UC04 - Atualizar Aluno

Ator Principal: Administrador

Descrição: Este caso de uso permite ao administrador atualizar os dados de um aluno existente.

Fluxo Principal:

  1. O administrador envia uma requisição PUT para o endpoint /api/alunos/{id} com os novos dados do aluno.
  2. O sistema valida os dados recebidos.
  3. O sistema atualiza os dados do aluno no banco de dados.
  4. O sistema retorna os dados atualizados do aluno com status 200 (OK).

Fluxo Alternativo:

  1. Se os dados forem inválidos, o sistema retorna uma mensagem de erro com status 400 (Bad Request).
  2. Se o aluno não for encontrado, o sistema retorna uma mensagem de erro com status 404 (Not Found).
  3. Se o email já estiver cadastrado para outro aluno, o sistema retorna uma mensagem de erro com status 409 (Conflict).
UC05 - Excluir Aluno

Ator Principal: Administrador

Descrição: Este caso de uso permite ao administrador excluir um aluno do sistema.

Fluxo Principal:

  1. O administrador envia uma requisição DELETE para o endpoint /api/alunos/{id}.
  2. O sistema busca o aluno com o ID especificado no banco de dados.
  3. O sistema exclui o aluno do banco de dados.
  4. O sistema retorna uma mensagem de sucesso com status 200 (OK).

Fluxo Alternativo:

  1. Se o aluno não for encontrado, o sistema retorna uma mensagem de erro com status 404 (Not Found).

Restrições e Limitações

As seguintes restrições e limitações se aplicam ao desenvolvimento do sistema:

  • O sistema deve ser desenvolvido utilizando Node.js e Express.
  • O banco de dados deve ser MySQL.
  • O sistema deve seguir o padrão de arquitetura MVC.
  • O sistema deve ser desenvolvido como uma API RESTful.
  • O sistema deve ser documentado utilizando comentários no código e README.
  • O sistema deve ser versionado utilizando Git.
